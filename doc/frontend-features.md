# 前端功能说明

本文档描述 `frontend/` 目录下手机网页的当前功能、每个按钮的用途，以及各处交互背后的设计意图。

---

## 1. 前端的整体目标

这个前端不是一个传统“表单页面”，而是一个尽可能接近**手机原生便签 + 输入法面板**体验的网页输入器。

它的核心目标是：

- 让用户在手机上专注输入
- 最大化复用手机输入法，尤其是语音输入
- 减少手指离开键盘区域的次数
- 在需要时提供少量但高频的快捷操作
- 让发送动作尽可能稳定，不打断输入法状态

---

## 2. 页面结构总览

页面主要由三部分组成：

1. **主输入区 `Composer`**
   - 占据页面主要空间
   - 用于输入、编辑、语音输入后的结果确认

2. **底部悬浮操作栏 `Dock`**
   - 放置常用快捷动作
   - 包括设置、复制、粘贴、Tab、换行、退格、发送

3. **设置弹层 `SettingsModal`**
   - 修改主题
   - 修改回车行为
   - 控制悬浮按钮显隐
   - 查看并复用最近发送记录

---

## 3. 主输入区（Composer）

文件：`frontend/src/components/Composer.tsx`

### 3.1 输入控件类型

当前主输入区是一个 **`textarea`**。

这样设计的原因是：

- 支持多行文本
- 更适合语音输入后的长文本整理
- 可以让用户手动插入换行
- 更像“正文编辑区域”，而不是单行命令输入框

### 3.2 自动聚焦

页面加载后，输入框会自动聚焦。

此外，当页面从后台切回前台时，也会再次尝试聚焦。

设计意图：

- 减少用户进入页面后的额外点击
- 保证键盘和输入法尽快进入工作状态
- 在手机端切出再回来时，尽量恢复“继续输入”的状态

### 3.3 点击空白区域自动回到输入框

在 `App.tsx` 中，页面有一个全局点击逻辑：

- 如果点击的不是 Dock
- 不是弹层
- 也不是输入框本身

就会自动把焦点拉回输入框。

设计意图：

- 保持页面的“沉浸式输入器”特性
- 避免用户点到空白后丢失输入焦点
- 降低键盘被意外收起的概率

### 3.4 自动高度调整

输入框会随着内容增长自动增高。

设计意图：

- 不出现固定高度小框
- 让多段文字更接近自然书写体验
- 减少内部滚动条带来的割裂感

### 3.5 输入框为空时的行为

当输入框为空时：

- 按键盘上的 **Backspace**：不会删除网页里的内容（因为本来就没有），而是直接向电脑发送一个 `backspace`
- 提交动作（例如触发发送逻辑）时：不会发送空文本，而是直接向电脑发送一个 `enter`

设计意图：

- 让网页在“没文字可发”时，退化成一个远程键盘动作面板
- 避免空文本请求
- 让回车和退格在“空输入状态”下仍然有价值

### 3.6 输入框有文字时的行为

当输入框里有内容时：

- 发送按钮会把整段文本发给电脑
- 发送成功后，文本会清空
- 同时写入历史记录
- 焦点会回到输入框，方便继续下一段输入

设计意图：

- 保持“输入一段 -> 发送 -> 继续输入下一段”的循环流畅
- 让历史记录可复用

### 3.7 回车键行为（可配置）

回车键有两种模式，由设置决定：

#### 模式 A：直接发送
- 当输入框有内容时，按回车会触发发送
- 当输入框为空时，提交会向电脑发送 `enter`

#### 模式 B：换行编写
- 当输入框有内容时，按回车会保留原生换行
- 用户可以继续编辑多行文本
- 真正发送依赖底部发送按钮

设计意图：

- 有些用户想“回车即发”
- 有些用户想先写好多行再发
- 两种习惯都应该被支持

### 3.8 `enterkeyhint` 的动态切换

输入框会动态调整 `enterkeyhint`：

- 有文字且当前不是“换行模式”时：显示为 `send`
- 否则显示为 `enter`

它只会在：

- `keyup`
- `compositionend`

之后更新。

设计意图：

- 让手机键盘更贴合当前操作语义
- 尽量避免在组合输入/语音输入过程中频繁修改属性，打断输入法状态

这对移动端输入法、连续语音识别、中文组合输入尤其重要。

### 3.9 组合输入保护

输入框在组合输入（IME composing）期间，会避免把 Enter 等按键逻辑当成普通提交处理。

设计意图：

- 避免中文输入法选词过程中被误判为“提交”
- 避免语音输入/拼音输入中途被打断

## 4. 底部悬浮操作栏（Dock）

文件：`frontend/src/components/Dock.tsx`

Dock 是整个页面的高频操作区，采用底部悬浮的胶囊式布局。

### 4.1 整体布局设计

Dock 固定在底部，并会根据 `visualViewport` 动态抬升，避免被软键盘遮住。

设计意图：

- 让快捷按钮始终保持在拇指可触达区域
- 当键盘弹出时，操作栏自动“浮到键盘上方”
- 不要求用户关闭键盘再点按钮

### 4.2 为什么使用 `pointerdown`

很多按钮会优先在 `pointerdown` 阶段处理，并调用 `preventDefault()`。

设计意图：

- 尽量避免点击按钮时输入框失焦
- 避免软键盘闪收再弹出
- 保持输入流畅，尤其是连续听写后的发送动作

## 5. Dock 中每一个按钮的作用

下面按当前从左到右的顺序说明。

### 5.1 设置按钮

图标：齿轮

作用：

- 打开设置与历史弹层

交互特点：

- 点击后弹出底部模态面板
- 使用 `pointerdown` 防止点击时导致输入框失焦抖动

设计意图：

- 把低频设置放到二级面板
- 保持主界面纯净

---

### 5.2 复制按钮

图标：两个重叠矩形

作用：

- 向电脑发送一次 `copy`
- 后端当前实现为模拟 `Ctrl+C`

适合场景：

- 远程复制电脑当前选中文本

交互特点：

- 单次触发按钮
- 不支持长按连发
- 触发时有轻微震动反馈

---

### 5.3 粘贴按钮

图标：剪贴板

作用：

- 向电脑发送一次 `paste`
- 后端当前实现为模拟 `Ctrl+V`

适合场景：

- 把电脑当前剪贴板内容粘贴到焦点位置

交互特点：

- 单次触发按钮
- 不支持长按连发
- 触发时有轻微震动反馈

---

### 5.4 Tab 按钮

图标：向右箭头 + 左侧竖线

作用：

- 向电脑发送 `tab`

适合场景：

- 缩进
- 表单跳转
- 编辑器内快速移动焦点

交互特点：

- 支持**长按连续触发**
- 长按 450ms 后开始自动重复
- 重复频率约为每 100ms 一次
- 会显示连击计数浮层，如 `x3`、`x8`

设计意图：

- 单次点按满足常规缩进
- 长按可用于更高频的跳转/缩进场景

---

### 5.5 换行按钮

图标：回折箭头

作用：

- 向电脑发送 `newline`
- 后端当前实现不是模拟 Enter，而是**粘贴一个 `\n` 字符**

适合场景：

- 需要“插入文本换行”而不触发表单提交时
- 在某些编辑器/文本框中插入纯文本换行

交互特点：

- 支持长按连续触发
- 同样带有计数浮层

设计意图：

- 把“动作回车”和“文本换行”区分开
- 减少某些场景下 Enter 被宿主应用解释成提交/确认的副作用

---

### 5.6 退格按钮

图标：退格键形状

作用：

- 向电脑发送 `backspace`

适合场景：

- 删除电脑焦点处的内容
- 快速修正远端文本

交互特点：

- 支持长按连续触发
- 同样有重复计数浮层

设计意图：

- 退格是高频动作，必须支持长按
- 计数反馈可以帮助用户感知已经连发了多少次

---

### 5.7 分隔线

作用：

- 将“辅助动作键”与“主发送按钮”视觉上分组

设计意图：

- 让发送按钮成为右侧主操作
- 提高主次分层清晰度

---

### 5.8 发送按钮

图标：向上箭头

作用：

- 当输入框有文字时：发送当前文本到电脑
- 当输入框没有文字时：当前按钮视觉上会变灰，且不会触发发送逻辑

交互特点：

- 有文字时可点，透明度为 1
- 无文字时显示为弱化状态，透明度降低，鼠标样式也变成不可用态
- 发送成功后按钮会有一次成功动画

设计意图：

- 发送是页面的核心主动作，必须视觉突出
- 但空文本发送意义不大，因此主发送按钮在空状态下不参与触发
- “空文本时发 Enter”的逻辑保留在输入提交路径中，而不是这个显式发送按钮里

## 6. 连续触发与计数反馈

文件：`frontend/src/hooks/useContinuousTrigger.ts`

这个 hook 被 Dock 中的连续动作按钮复用，例如：

- Tab
- Newline
- Backspace

### 6.1 单次触发按钮

例如 Copy / Paste：

- 按下立即触发一次
- 不进入长按重复模式
- 触发后短时间内锁住，避免重复误触

### 6.2 连续触发按钮

例如 Tab / Newline / Backspace：

- 按下立即触发第一次
- 长按 450ms 后开始连续发送
- 每 100ms 自动再触发一次
- 松开时停止

### 6.3 计数气泡

连续按钮上方会出现计数气泡：

- `x2`
- `x5`
- `x12`

设计意图：

- 给用户明确反馈：动作已经连发了多少次
- 减少“我到底删了几下 / 缩进了几次”的不确定感

### 6.4 震动反馈

每次触发会调用 `navigator.vibrate()`（如果设备支持）：

- 初始触发的震动更明显
- 连发时震动更轻

设计意图：

- 增加触觉确认
- 尤其适合移动端盲操

## 7. 设置弹层（SettingsModal）

文件：`frontend/src/components/SettingsModal.tsx`

这是一个底部滑出的 `dialog` 模态面板。

### 7.1 打开方式

- 点击 Dock 左侧齿轮按钮打开

### 7.2 关闭方式

- 点击右上角关闭按钮
- 点击弹层外部背景区域
- `dialog` 关闭时同步触发外层状态更新

设计意图：

- 尽量符合移动端底部抽屉/Action Sheet 习惯

## 8. 设置项详解

### 8.1 界面外观

可选项：

- 系统
- 浅色
- 深色

作用：

- `system`：跟随系统 `prefers-color-scheme`
- `light`：强制浅色
- `dark`：强制深色

实现方式：

- 把选择写入本地偏好
- 通过 `data-theme` 控制 CSS 变量

设计意图：

- 给用户保留手动控制视觉风格的能力
- 又不丢掉系统联动

---

### 8.2 键盘回车键功能

可选项：

- 直接发送
- 换行编写

作用：

- 决定输入框内按回车时是提交还是插入换行

设计意图：

- 适配不同写作习惯
- 兼顾“短句立即发送”和“长文本分段编辑”两类使用方式

---

### 8.3 悬浮岛按键显示

可控制显隐的按钮有：

- 复制
- 粘贴
- Tab 缩进
- 插入换行
- 实体退格

作用：

- 用户可以按自己的习惯删掉不需要的按钮
- 隐藏后主界面会更简洁

设计意图：

- 并不是每个人都需要全部快捷键
- Dock 空间珍贵，应允许按需裁剪

---

### 8.4 最近发送记录

作用：

- 展示最近发送过的文本
- 最多保存 50 条
- 点击某一条会把内容重新放回输入框

显示内容：

- 发送时间
- 文本内容（保留换行展示）

交互特点：

- 点击历史项后，会关闭弹层
- 把文本放回输入框
- 稍后自动重新聚焦输入框

设计意图：

- 很多发送内容会重复使用
- 让“复发一段旧文本”成本更低

### 8.5 清空全部

作用：

- 清空本地保存的全部历史记录

设计意图：

- 给用户保留对本地数据的控制权

## 9. 偏好与本地存储

文件：`frontend/src/hooks/usePreferences.ts`

当前前端会把这些内容保存在 `localStorage`：

- 主题设置
- 回车行为
- Dock 按钮显隐
- 最近发送历史

### 9.1 历史去重

新增历史时会：

- 去掉内容完全相同的旧项
- 把最新一条插到最前面
- 总数最多保留 50 条

设计意图：

- 保持历史简洁
- 优先保留最近且常用的内容

## 10. 视口与软键盘适配

文件：`frontend/src/hooks/useViewportOffset.ts`

前端会监听 `visualViewport` 的：

- `resize`
- `scroll`

并计算当前软键盘占据的底部高度，写入 CSS 变量 `--keyboard-offset`。

### 10.1 作用

这个变量会影响 Dock 的 `bottom`：

- 键盘收起时，Dock 靠近底部安全区
- 键盘弹出时，Dock 自动上移到键盘上方

设计意图：

- 保持按钮可点
- 减少“按钮被键盘遮住”的问题
- 让整个页面更像原生移动应用

## 11. API 与 endpoint 设计

文件：`frontend/src/utils/api.ts`

### 11.1 默认行为

默认情况下，前端请求：

- `/api/type-text`
- `/api/press-key`

在开发模式下，Vite 会把这些请求代理到 Rust。

### 11.2 自定义 endpoint

前端还支持两种自定义 endpoint 的方式：

1. URL 参数：`?endpoint=...`
2. `localStorage` 持久化保存

作用：

- 允许网页连接到一个指定的后端地址
- 给未来的“手动指定目标机器”留下扩展空间

### 11.3 `press-key` 地址的推导方式

前端会先拿到 `type-text` 的 endpoint，再自动把路径末尾替换成 `/press-key`。

设计意图：

- 减少重复配置
- 只要知道主文本接口，动作接口可以自动推导

## 12. 当前页面的视觉设计思路

从样式上看，这个页面不是传统表单，而是一个：

- 全屏正文输入区
- 底部玻璃感悬浮岛
- 底部上滑设置面板

### 12.1 主输入区

- 背景纯净
- 字体较大
- 行距较松
- 更接近写正文，而不是填表

### 12.2 Dock

- 毛玻璃风格
- 胶囊形态
- 主发送按钮反差更强

### 12.3 主题系统

通过 CSS 变量同时支持：

- 浅色
- 深色
- 跟随系统

整体目标是：

> 让输入成为页面唯一主角，其余操作尽量退到边缘，但又随时可达。

## 13. 一个完整的典型交互流程

以“用户用语音输入一段文字并发送”为例：

1. 打开网页
2. 输入框自动聚焦，手机键盘弹出
3. 用户用手机输入法进行语音输入
4. 文字流式出现在输入框里
5. 用户等待输入法做最终整理
6. 用户点击右下角发送按钮
7. 前端调用 `/api/type-text`
8. Rust 把文本写入电脑剪贴板并模拟粘贴
9. 发送成功后：
   - 输入框清空
   - 文本进入历史记录
   - 输入框重新聚焦
   - 发送按钮出现一次成功反馈

这条路径体现了整个前端的设计核心：

- 保持焦点
- 保持键盘
- 保持输入连续性
- 发送后立刻准备下一轮输入

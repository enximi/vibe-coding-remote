# 前端功能说明

本文档描述当前共享前端 UI 的功能、每个按钮的用途，以及各处交互背后的设计意图。

需要先区分两层：

- `packages/app`：真正的前端产品逻辑和 UI
- `apps/web`：Web 壳与平台桥接

下面提到的大多数功能都位于 `packages/app`。

## 1. 前端的整体目标

这个前端不是传统表单页，而是一个尽可能接近“手机原生便签 + 输入法面板”体验的网页输入器。

它的核心目标是：

- 让用户在手机上专注输入
- 最大化复用手机输入法，尤其是语音输入
- 减少手指离开键盘区域的次数
- 在需要时提供少量但高频的快捷操作
- 让发送动作尽可能稳定，不打断输入法状态

## 2. 页面结构总览

页面主要由三部分组成：

1. 主输入区 `Composer`
2. 底部悬浮操作栏 `Dock`
3. 设置与历史面板 `SettingsModal`

## 3. 主输入区（Composer）

文件：`packages/app/src/components/Composer.tsx`

### 3.1 输入控件类型

当前主输入区是一个 `textarea`。

这样设计的原因是：

- 支持多行文本
- 更适合语音输入后的长文本整理
- 可以让用户手动插入换行
- 更像正文编辑区域，而不是单行命令输入框

### 3.2 自动聚焦

页面加载后，输入框会自动聚焦。

此外，当页面从后台切回前台时，也会再次尝试聚焦。

设计意图：

- 减少用户进入页面后的额外点击
- 保证键盘和输入法尽快进入工作状态
- 切出再回来时尽量恢复“继续输入”的状态

### 3.3 点击空白区域自动回到输入框

在 `packages/app/src/App.tsx` 里，页面有一个全局点击逻辑：

- 如果点击的不是 Dock
- 不是弹层
- 也不是输入框本身

就会自动把焦点拉回输入框。

设计意图：

- 保持页面的沉浸式输入器特性
- 避免用户点到空白后丢失输入焦点
- 降低键盘被意外收起的概率

### 3.4 自动高度调整

输入框会随着内容增长自动增高。

设计意图：

- 避免固定高度小框
- 让多段文字更接近自然书写体验
- 减少内部滚动条的割裂感

### 3.5 草稿自动保存与恢复

当前输入内容会在本地自动保存为草稿。

当页面刷新、切到后台后被浏览器回收，或者意外重新加载时，草稿会自动恢复。

恢复后：

- 文本内容会回填
- 光标会自动放到文本末尾

设计意图：

- 防止长文本在移动端意外丢失
- 让恢复后的继续输入成本最低

### 3.6 输入框为空时的行为

当输入框为空时：

- 按键盘上的 Backspace：不会删除网页里的内容，而是直接向电脑发送一个 `backspace`
- 触发提交动作时：不会发送空文本，而是直接向电脑发送一个 `enter`

设计意图：

- 让网页在“没文字可发”时退化成远程动作面板
- 避免空文本请求
- 让回车和退格在空输入状态下仍然有价值

### 3.7 输入框有文字时的行为

当输入框里有内容时：

- 发送按钮会把整段文本发给电脑
- 发送成功后，文本会清空
- 文本会写入历史记录
- 焦点会回到输入框，方便继续下一轮输入

### 3.8 回车键行为（可配置）

回车键有两种模式，由设置决定：

#### 模式 A：直接发送

- 当输入框有内容时，按回车会触发发送
- 当输入框为空时，提交会向电脑发送 `enter`

#### 模式 B：换行编写

- 当输入框有内容时，按回车会保留原生换行
- 用户可以继续编辑多行文本
- 真正发送依赖底部发送按钮

### 3.9 `enterkeyhint` 的动态切换

输入框会动态调整 `enterkeyhint`：

- 有文字且当前不是“换行模式”时：显示为 `send`
- 否则显示为 `enter`

它只会在：

- `keyup`
- `compositionend`

之后更新。

设计意图：

- 让手机键盘更贴合当前操作语义
- 尽量避免在组合输入或语音输入过程中频繁修改属性，打断输入法状态

### 3.10 组合输入保护

输入框在组合输入（IME composing）期间，会避免把 Enter 等按键逻辑当成普通提交处理。

设计意图：

- 避免中文输入法选词过程中被误判为提交
- 避免语音输入 / 拼音输入中途被打断

## 4. 底部悬浮操作栏（Dock）

文件：`packages/app/src/components/Dock.tsx`

Dock 是整个页面的高频操作区，采用底部悬浮胶囊布局。

### 4.1 整体布局设计

Dock 固定在底部，并会根据 `visualViewport` 动态抬升，避免被软键盘遮住。

### 4.2 为什么使用 `pointerdown`

很多按钮会优先在 `pointerdown` 阶段处理，并调用 `preventDefault()`。

设计意图：

- 避免点击按钮时输入框失焦
- 避免软键盘闪收再弹出
- 保持输入流畅，尤其是连续听写后的发送动作

## 5. Dock 中每一个按钮的作用

### 5.1 设置按钮

作用：

- 打开设置与历史面板

### 5.2 复制按钮

作用：

- 向电脑发送一次 `copy`
- 后端当前实现为模拟 `Ctrl+C`

### 5.3 粘贴按钮

作用：

- 向电脑发送一次 `paste`
- 后端当前实现为模拟 `Ctrl+V`

### 5.4 Tab 按钮

作用：

- 向电脑发送 `tab`

交互特点：

- 支持长按连续触发
- 长按 450ms 后开始自动重复
- 重复频率约为每 100ms 一次
- 显示连击计数浮层

### 5.5 换行按钮

作用：

- 向电脑发送 `newline`
- 后端当前实现不是模拟 Enter，而是粘贴一个 `\n`

设计意图：

- 把“动作回车”和“文本换行”区分开
- 降低宿主应用把 Enter 当作提交的副作用

### 5.6 退格按钮

作用：

- 向电脑发送 `backspace`

交互特点：

- 支持长按连续触发
- 有连击计数反馈

### 5.7 发送按钮

作用：

- 当输入框有文字时：发送当前文本到电脑
- 当输入框没有文字时：视觉上变灰，不参与发送逻辑

设计意图：

- 发送是页面的核心主动作，必须视觉突出
- “空文本时发 Enter”的逻辑保留在输入提交路径里，而不是这个显式发送按钮里

## 6. 连续触发与计数反馈

文件：`packages/app/src/hooks/useContinuousTrigger.ts`

这个 hook 被 Dock 中的连续动作按钮复用，例如：

- Tab
- Newline
- Backspace

### 6.1 单次触发按钮

例如 Copy / Paste：

- 按下立即触发一次
- 不进入长按重复模式
- 触发后短时间内锁住，避免重复误触

### 6.2 连续触发按钮

例如 Tab / Newline / Backspace：

- 按下立即触发第一次
- 长按 450ms 后开始连续发送
- 每 100ms 自动再触发一次
- 松开时停止

### 6.3 计数气泡

连续按钮上方会出现计数气泡，例如 `x2`、`x5`、`x12`。

### 6.4 震动反馈

Web 壳会通过 `navigator.vibrate()` 提供触感反馈；Tauri 壳当前没有 haptic 实现。

## 7. 设置面板（SettingsModal）

文件：`packages/app/src/components/SettingsModal.tsx`

这是一个底部滑出的模态面板。

### 7.1 打开方式

- 点击 Dock 左侧设置按钮

### 7.2 关闭方式

- 点击关闭按钮
- 点击弹层外部区域
- 面板关闭时同步更新外层状态

## 8. 设置项详解

### 8.1 界面外观

可选项：

- 系统
- 浅色
- 深色

### 8.2 键盘回车键功能

可选项：

- 直接发送
- 换行编写

### 8.3 Dock 按键显隐

可控制显隐的按钮有：

- 复制
- 粘贴
- Tab 缩进
- 插入换行
- 实体退格

### 8.4 最近发送记录

作用：

- 展示最近发送过的文本
- 最多保存 50 条
- 点击某一条会把内容重新放回输入框

显示内容：

- 发送时间
- 文本内容（保留换行展示）

### 8.5 清空全部

作用：

- 清空本地保存的全部历史记录

## 9. 偏好与本地存储

文件：`packages/app/src/hooks/usePreferences.ts`

当前前端会把这些内容保存在 `localStorage`：

- 主题设置
- 回车行为
- Dock 按钮显隐
- 最近发送历史

草稿则由：

- `packages/app/src/features/editor/draft.ts`

单独管理。

## 10. 视口与软键盘适配

文件：`packages/app/src/hooks/useViewportOffset.ts`

前端会监听 `visualViewport` 的：

- `resize`
- `scroll`

并计算当前软键盘占据的底部高度，写入 CSS 变量 `--keyboard-offset`。

这个变量会影响 Dock 的 `bottom`，让它在键盘弹出时自动上移到可操作区域。

## 11. 平台桥接与 API

### Web 壳桥接

文件：`apps/web/src/platform/bridge.ts`

默认情况下，Web 壳请求：

- `/api/type-text`
- `/api/press-key`

开发模式下，Vite 会把这些请求代理到 Rust。

它还支持两种自定义 endpoint 的方式：

1. URL 参数：`?endpoint=...`
2. `localStorage` 持久化保存

### Tauri 壳桥接

文件：`apps/tauri/src/platform/bridge.ts`

当前 Tauri 壳直接请求：

- `http://127.0.0.1:8765/api/type-text`
- `http://127.0.0.1:8765/api/press-key`

这让两种壳共用同一套后端 API 语义。

## 12. 一个典型交互流程

以“用户用语音输入一段文字并发送”为例：

1. 打开页面
2. 输入框自动聚焦，手机键盘弹出
3. 用户用手机输入法进行语音输入
4. 文字流式出现在输入框里
5. 用户等待输入法做最终整理
6. 用户点击右下角发送按钮
7. 前端调用 `bridge.sendText()`
8. Rust 把文本写入电脑剪贴板并模拟粘贴
9. 发送成功后：
   - 输入框清空
   - 文本进入历史记录
   - 输入框重新聚焦
   - 发送按钮出现一次成功反馈

这条路径体现了整个前端的设计核心：

- 保持焦点
- 保持键盘
- 保持输入连续性
- 发送后立刻准备下一轮输入

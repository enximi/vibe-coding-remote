# PRD

## 产品概述

声桥是一个把手机上的文字输入能力，尤其是语音输入能力，低摩擦地桥接到电脑当前光标位置的小工具。

它的核心价值不是“手机能给电脑发一段文字”，而是让用户在电脑上的实际工作流中，用手机上更成熟、更准确的输入方式，替代电脑端不理想的语音输入体验。

## 背景与问题

本产品的直接动机来自 AI 编程场景，尤其是使用 Codex、Claude Code 等需要持续输入自然语言指令的工作流。

在这一类工作中，用户经常需要：

- 快速描述需求、问题、修改方向
- 连续多轮输入较长的自然语言
- 在 AI 工作期间继续准备下一轮输入

相比键盘打字，语音输入在很多场景下效率更高、更顺畅，因此自然会引出“能否直接用语音来驱动 AI 编程”的需求。

市场上已经存在一些电脑端语音输入方案，但整体体验仍有明显缺口：

- 豆包电脑版支持快捷键开始语音输入，结束后通过 `Enter` 插入文本、通过 `Esc` 取消
- 微信近期也提供了快捷键驱动的语音输入能力
- 一些第三方产品会调用大模型或语音识别 API 来实现电脑端听写

但这些方案普遍存在至少一类问题：

- 识别准确率不如手机版豆包输入法
- 对语气词、常用词和专业词的处理不理想
- 文字插入到电脑当前光标处的成功率不稳定
- 实际交互虽然“能用”，但不够流畅

在主观体验上，手机端豆包输入法的语音输入效果明显优于电脑端同类方案。无论是识别质量、整理效果还是使用过程中的顺滑程度，手机输入法都更接近理想状态。

除了识别效果本身之外，手机方案还有一些电脑方案难以替代的现实优势：

- 某些场景下，用户更适合对着手机说话，而不适合直接对着电脑说话
- 有些工作环境中的电脑并没有可用麦克风，或麦克风条件较差
- 在安静环境里，用户往往更愿意用较低音量对着手机说话
- 手机版豆包输入法在用户音量较低时，仍然能保持较高的识别准确率

这意味着，手机不仅是“另一个输入终端”，而且在很多真实场景里，本身就是更自然、更容易被接受的语音输入载体。

因此，本产品的核心判断是：

> 与其继续寻找电脑端“勉强可用”的语音输入，不如直接复用手机上已经成熟的语音输入法能力，再把结果稳定地送到电脑光标处。

## 产品目标

- 复用手机端优秀输入法，尤其是豆包输入法的语音输入能力
- 把识别结果稳定、正确地插入到电脑当前光标所在位置
- 让整个过程足够流畅，能够成为 AI 编程中的高频日常工具
- 尽量减少用户在手机上的额外操作与手指移动范围
- 让高频操作尽可能在手机输入法键盘区域内完成，而不是跳出到复杂的网页控件

本产品追求的不只是“可用”，而是“在高频工作流中几乎不打断思路”的使用体验。

## 非目标

- 不以自研一套全新的语音识别能力为目标
- 不追求替代手机输入法本身的编辑、听写与候选词能力
- 不优先解决所有桌面应用中的复杂富文本兼容性问题
- 不将“功能堆砌”作为目标，优先保证高频路径的顺滑与可靠

## 目标用户

- 经常使用 AI 编程工具，需要连续输入自然语言的人
- 对语音输入效率敏感，希望减少打字负担的人
- 电脑端语音输入体验不理想，但手机端输入法体验较好的人
- 电脑麦克风条件一般，或不希望依赖电脑收音设备的人
- 在安静环境中更适合低声对着手机说话，而不想直接对着电脑说话的人

## 核心使用场景

典型场景是 AI 编程中的一轮指令输入：

1. 用户在电脑上把光标放到目标输入框中
2. 用户在手机上唤出豆包输入法
3. 用户按住空格进行语音输入
4. 输入法完成流式识别与最终整理
5. 用户直接在键盘区域内点击或滑动到“发送”
6. 识别结果被插入到电脑当前光标处
7. 如果需要提交给 AI，用户再通过手机侧操作触发电脑上的 `Enter`
8. AI 开始工作，用户可以继续准备下一轮输入

这一场景下，理想状态是用户的手指尽量不离开输入法键盘区域，高频动作尽量在同一片操作区域内完成。

另一个重要场景是：

1. 用户所处环境比较安静，不希望明显地对着电脑说话
2. 用户更容易接受把手机拿近一点，用较低音量说话
3. 手机输入法仍然可以稳定识别
4. 最终结果再被可靠地送到电脑

这个场景说明，本产品不只是“把手机输入法搬到电脑上”，也在解决“哪些设备更适合被当作语音入口”的问题。

## 产品范围

当前产品范围聚焦于“手机语音输入到电脑文本插入”的窄而深体验：

- 手机侧提供尽可能轻量、沉浸的输入界面
- 电脑侧接收文本或按键信号，并作用到当前焦点位置
- 补齐发送、回车、退格、换行等高频编辑动作
- 围绕 AI 编程场景打磨实际可用的交互细节

## 版本范围

当前阶段以高频主路径为中心：

- 手机上输入或语音输入一段内容
- 将结果发送到电脑
- 在空输入状态下也能远程触发 `Enter`、`Backspace`
- 通过少量悬浮按钮补充换行、退格等动作

## 风险与限制

- 手机输入法的能力高度依赖系统与具体输入法产品，本产品无法直接控制其内部行为
- 不同桌面应用对文本插入、回车和换行的解释可能不同
- 用户想要复用“手机输入法的强项”，就天然会受限于手机端交互模型
- 某些优势来自手机本身作为语音入口的形态，因此完全替换为电脑端方案未必能保留同样体验
- 如果要进一步实现“手完全不离开电脑键盘”，产品形态可能需要转向“电脑发起、手机只负责识别”的新方案

## 开放问题

- 是否要继续沿“手机主动输入”这条路线深挖，还是规划“电脑快捷键发起”的下一代交互模型
- 是否需要围绕特定输入法做更深的适配或优化
- 是否要在未来引入 Android 原生 App 作为比 Web 更稳定的手机端形态

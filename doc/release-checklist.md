# 发布前检查清单

这份清单面向“准备把声桥发布到 GitHub 并对外开源”的场景。

---

## 必做项

### 1. 选择许可证

这是最重要的一项。

当前仓库还没有正式的 `LICENSE` 文件。

如果你准备对外开源，建议先明确：

- MIT
- Apache-2.0
- GPL-3.0
- 其他你想采用的许可证

没有许可证时，别人通常**不能默认自由使用、修改和分发你的代码**。

### 2. 确认仓库中没有误提交的本地文件

重点检查：

- `node_modules`
- `dist`
- `target`
- 本地日志
- 私人测试数据
- 调试脚本
- 截图中是否包含隐私信息

### 3. 确认 README 可独立读懂

从一个陌生开发者的视角检查：

- 项目是做什么的
- 为什么值得用
- 怎么跑起来
- 平台限制是什么
- 当前功能边界是什么

### 4. 确认开发与生产说明清楚

目前项目有两种访问方式：

- 开发：访问 `5173`
- 生产：访问 `8765`

这个区别必须写清楚，否则别人第一次运行时很容易混淆。

### 5. 确认基础检查能通过

建议至少确认：

#### Rust

```powershell
cargo fmt
cargo check
```

#### Frontend

```powershell
cd frontend
bun run lint
bun run build
```

---

## 推荐补充项

### 6. 增加仓库截图或演示 GIF

这个项目非常依赖“交互感受”，单靠文字不如截图或短 GIF 直观。

推荐至少准备：

- 手机输入界面截图
- 设置面板截图
- 一段语音输入 -> 电脑落字的演示 GIF

### 7. 增加一个简短的架构图

哪怕只是 README 里的 ASCII 图，也会帮助别人快速理解：

```text
Phone Browser -> Rust Local Server -> Focused Desktop App
```

### 8. 考虑是否需要 CONTRIBUTING.md

如果你愿意接受 issue / PR，建议保留并完善贡献说明。

当前仓库已经有基础版，可以根据你自己的协作偏好继续补充。

### 9. 统一命名和措辞

开源前建议再扫一遍：

- 仓库标题
- README 中英文名称
- 前端页面 title
- 提交信息风格
- 文档中的术语

让外部读者看到的是同一套语言系统。

### 10. 明确当前限制

建议在 README 或文档中明确写出：

- 当前桌面端偏 Windows
- 文本输入依赖剪贴板 + 粘贴
- 当前不包含设备认证
- 当前主要依赖局域网环境

这样可以减少错误预期。

---

## 发布当天建议流程

1. 运行一遍检查命令
2. 最后看一遍 `git status`
3. 补充或确认 `LICENSE`
4. 检查 README 和 docs 链接是否可用
5. 推送到 GitHub
6. 设置仓库描述、Topics、首页链接
7. 如有演示视频或截图，补到 README

---

## 可选的 GitHub 仓库配置

发布后你还可以在 GitHub 页面补：

- 仓库 Description
- Topics（如 `rust`、`react`、`windows`、`dictation`、`input-method`）
- Website（如果以后有主页）
- Issue 模板
- Discussion

## 一句话提醒

如果只留一件事最后手动确认，那就是：

> **先选好许可证，再公开发布。**

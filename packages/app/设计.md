# App 设计

## 1. 定位

`packages/app` 是 Voice Bridge 的通用前端应用主体。

它负责：

- 编辑区 UI
- 快捷动作 UI
- 设置与历史记录 UI
- 输入交互规则
- 页面状态切换

它不负责：

- Web / Tauri 的平台启动方式
- server 的实际 HTTP 实现
- 平台专属能力的底层接入

---

## 2. 核心目标

这个应用的目标不是“在手机上再做一个聊天框”，而是：

> 让用户在手机上尽量不离开输入法键盘区域，就能把文字或动作高效发送到电脑当前光标处。

设计原则：

1. 高频操作优先在键盘区域内完成
2. 记事本式编辑体验优先于复杂界面
3. 发送前可编辑、可修改、可暂存
4. 连接状态必须清楚
5. 稳定性优先于花哨交互

---

## 3. 页面结构

### 3.1 主工作区

页面主体是一个类似记事本正文的全屏编辑区域。

要求：

- 默认占据页面主要视觉空间
- 进入页面后尽量自动聚焦
- 用户可以直接打字、语音输入、编辑已有文本
- 编辑区为空时显示引导说明

这个区域是整个应用的核心。

---

### 3.2 快捷动作区

页面需要提供快捷动作按钮，用户可通过偏好设置决定显示哪些按钮。

支持的独立动作：

- 发送 `enter`
- 发送 `tab`
- 发送 `backspace`
- 发送 `ctrl+c`
- 发送 `ctrl+v`
- 发送 `shift+tab`
- 粘贴换行符 `\n`

另外还需要始终提供：

- 进入更多界面的按钮
- 发送按钮

要求：

- 每个动作都有独立按钮能力
- 用户可以自定义显示 / 隐藏
- 高频按钮优先暴露在主界面
- 低频按钮可以进入更多界面后再使用
- 进入更多 / 设置面板的入口固定放在快捷动作区最左侧
- 发送按钮固定放在快捷动作区最右侧

这样底部区域形成稳定结构：

- 左侧：更多 / 设置
- 中间：快捷动作
- 右侧：发送

---

### 3.3 更多界面

更多界面承载非高频功能和全局设置。

形式：

- 使用 `bottom sheet`
- 从主界面唤起
- 在异常状态下可直接自动展开
- 正常状态下由底部快捷动作区最左侧入口按钮打开

内容包括：

- 地址设置
- token 设置
- 主题设置
- 键盘回车行为设置
- 历史记录
- 快捷动作显示配置

---

## 4. 页面状态

应用至少有以下几种状态：

### 4.1 未配置

条件：

- 没有地址
- 或没有 token

表现：

- 保持单主界面结构
- 直接展开更多 / 设置 `bottom sheet`
- 在地址、token 等相关设置项上提供轻量引导
- 不额外做独立错误页面
- 底部快捷动作区最左侧入口按钮可使用轻量强调样式，帮助用户理解修复入口位置

---

### 4.2 可工作

条件：

- 地址和 token 已配置
- 当前未发现连接错误

表现：

- 显示正常编辑区
- 显示快捷动作区
- 用户可直接输入、发送、查看历史

---

### 4.3 连接错误

条件：

- 地址无效
- server 不可达

表现：

- 保持单主界面结构
- 直接展开更多 / 设置 `bottom sheet`
- 在地址相关区域显示错误提示
- 引导用户检查地址或 server 是否启动
- 底部快捷动作区最左侧入口按钮可使用轻量强调样式

---

### 4.4 认证错误

条件：

- token 缺失
- token 错误

表现：

- 保持单主界面结构
- 直接展开更多 / 设置 `bottom sheet`
- 在 token 相关区域显示错误提示
- 引导用户检查 token
- 底部快捷动作区最左侧入口按钮可使用轻量强调样式

---

## 4.5 连接检查规则

前端需要明确区分三类问题：

- 未配置
- 地址 / server 可达性问题
- token 认证问题

检查规则如下。

### 检查前置条件

- 如果地址为空或 token 为空，直接进入“未配置”状态
- 此时不发起网络检查

### 地址检查

当地址和 token 都已填写后，前端先请求：

```http
GET /health
```

这个请求不带 token，只用于判断：

- 地址是否正确
- server 是否可达

结果映射：

- 请求失败、超时、网络错误：进入“连接错误”状态
- 返回非成功结果：进入“连接错误”状态
- 返回成功结果：进入下一步认证检查

### 认证检查

地址检查通过后，前端继续请求：

```http
GET /api/auth-check
Authorization: Bearer <token>
```

这个请求用于判断 token 是否正确，不产生任何输入副作用。

结果映射：

- 返回 `401`：进入“认证错误”状态
- 请求失败：进入“连接错误”状态
- 返回成功结果：进入“可工作”状态

### 检查触发时机

前端至少应在以下时机执行连接检查：

- 应用启动时
- 地址变更后
- token 变更后
- 用户从设置面板保存配置后
- 用户主动点击重试后

### UI 配合规则

- 检查进行中时，可进入“checking”这一内部过渡状态
- `checking` 状态不需要单独做页面，只需要轻量反馈即可
- 如果最终进入“连接错误”或“认证错误”，直接展开设置 `bottom sheet`
- 并在对应设置项附近显示错误提示

---

## 5. 编辑区行为

### 5.1 自动聚焦

目标：

- 用户进入应用或网页时，编辑区尽量自动聚焦

现实约束：

- Web 环境不一定能在无用户手势时稳定唤起系统键盘
- 因此设计目标应是“尽量自动聚焦”，允许必要时通过一次点击进入输入状态

---

### 5.2 草稿

要求：

- 编辑区内容需要自动保存为草稿
- 用户离开页面再回来时应自动恢复
- 恢复后光标应位于文本末尾

---

### 5.3 历史记录回填

要求：

- 用户点击历史记录后，文本应被放回编辑区，光标应位于文本末尾
- 用户可以继续编辑
- 也可以直接发送

---

## 6. 键盘行为规则

这是整个产品的核心交互规则。

### 6.1 编辑区为空时

手机键盘回车键：

- 发送一次 `enter`

手机键盘删除键：

- 发送一次 `backspace`

此时编辑区本身没有内容，因此这两个按键都作用到电脑端。

---

### 6.2 编辑区非空时

手机键盘删除键：

- 只删除编辑区中的一个字符
- 不触发任何特殊远程动作

手机键盘回车键：

- 行为由用户设置决定
- 支持两种模式：
  - `send`
  - `newline`

#### `send` 模式

- 回车键触发发送
- 当前编辑区内容发送到电脑

#### `newline` 模式

- 回车键只在编辑区内插入换行

---

### 6.3 键盘按钮文案

手机键盘上的回车键应尽量根据当前行为实时显示为不同语义，例如：

- 发送
- 换行

但这里有一条重要原则：

> 不应为了追求按钮文案的完全实时正确，而破坏输入法稳定性或语音输入连续性。

稳定性优先于按钮样式。

---

## 7. 快捷动作规则

快捷动作区需要覆盖下面这些远程能力：

- `enter`
- `tab`
- `backspace`
- `ctrl+c`
- `ctrl+v`
- `shift+tab`
- 粘贴换行符 `\n`

其中：

- “发送 enter” 与 “粘贴换行符” 必须视为两个不同动作
- 前者是发送按键
- 后者是向目标粘贴文本 `\n`

用户可以通过偏好设置决定哪些按钮显示、哪些隐藏。

---

## 8. 发送行为

发送按钮只在应用处于可工作状态时显示。

规则：

- 未配置、连接错误、认证错误状态下，不应优先显示发送按钮
- 这些状态下应显示更合适的 UI，引导用户修复当前状态
- 编辑区为空时，发送按钮不发送文本
- 编辑区非空时，发送按钮将当前文本发送到电脑
- 发送成功后，文本进入历史记录
- 发送成功后，编辑区清空

---

## 9. 历史记录

历史记录需要支持：

- 查看历史文本
- 显示时间
- 点击回填编辑区
- 删除单条
- 清空全部
- 设置保存数量上限

历史记录是“可再次编辑的输入草稿池”，不是只读日志。

---

## 10. 偏好设置

偏好设置需要至少包含：

- server 地址
- server token
- 主题模式
  - 浅色
  - 深色
  - 跟随系统
- 键盘回车行为
  - 发送
  - 换行
- 快捷动作按钮显示配置
- 历史记录保存数量上限

---

## 11. 设计原则补充

### 11.1 主体必须简单

主界面不应像“控制台”或“配置面板”。

它首先应该像一个安静、可编辑、可输入的记事本。

---

### 11.2 高频操作优先

应该优先优化这些动作：

1. 聚焦输入
2. 语音输入
3. 发送文本
4. 发送回车
5. 删除

这几项决定日常体验。

---

### 11.3 状态必须明确

如果地址或 token 缺失、错误，界面必须明确告诉用户当前不能正常工作，而不是维持一个看似可用但实际不可用的空白编辑器。

---

## 12. 当前实现边界

当前阶段先聚焦：

- 局域网连接
- 编辑区体验
- 快捷动作体验
- 设置与历史记录

当前不在 `packages/app` 内讨论：

- 自动发现
- 配对流程
- 云端中继
- 多设备授权模型
